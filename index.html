1

Mac 개발 환경 기초 세팅

Homebrew · 터미널 도구 · Zsh · 버전 관리

개발 환경을 처음부터 제대로 세팅하면 이후 모든 작업이 훨씬 편해진다. 마치 요리하기 전에 재료를 준비하는 것처럼 — 도구를 갖추는 데 투자한 1~2시간이 이후 수백 시간을 아껴준다.

1.1 Homebrew — Mac용 앱 설치 관리자

Homebrew는 명령어 한 줄로 개발 도구를 설치하고 업데이트할 수 있는 도구다. 앱스토어처럼 편리하지만, 개발자에게 필요한 수천 가지 도구를 다룰 수 있다. Apple Silicon(M1/M2/M3/M4) Mac에서는 /opt/homebrew 경로에 설치된다.

# Homebrew 설치 (터미널에 이 명령어를 붙여넣고 실행)
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

# Apple Silicon Mac: 설치 후 PATH 등록 (.zshrc에 추가)
echo 'eval "$(/opt/homebrew/bin/brew shellenv)"' >> ~/.zshrc
source ~/.zshrc

설치 후 자주 쓰는 개발 도구들을 한 번에 설치한다:

brew install git node bun python fzf ripgrep jq bat eza fd
도구한 줄 설명대체하는 것
git코드 버전 관리 — 변경 내역을 저장하고 되돌릴 수 있는 도구—
nodeJavaScript를 터미널에서 실행하는 엔진—
bunnode보다 훨씬 빠른 JavaScript 실행 환경 + 패키지 설치 도구npm, yarn
pythonPython 3 — AI/데이터 스크립트에 필수—
fzf정확히 기억 안 나도 찾아주는 검색 도구 (히스토리, 파일 검색)—
ripgrep폴더 전체에서 특정 코드를 빠르게 찾는 검색 도구grep
jqJSON 데이터를 터미널에서 보기 좋게 출력하고 가공하는 도구—
bat파일 내용을 색깔 있게 보여주는 도구cat
eza파일 목록을 아이콘과 색상으로 보기 좋게 보여주는 도구ls
fd파일을 빠르고 직관적으로 찾는 검색 도구find

1.2 버전 관리 — mise

여러 프로젝트를 동시에 다루다 보면 "이 프로젝트는 Node 18, 저 프로젝트는 Node 22"처럼 서로 다른 버전이 필요한 경우가 생긴다. mise(미즈)는 Node, Python, Ruby 등 다양한 언어 버전을 프로젝트별로 자동으로 전환해주는 도구다.

# mise 설치
brew install mise

# 터미널 설정 파일(.zshrc)에 자동 활성화 등록
echo 'eval "$(mise activate zsh)"' >> ~/.zshrc
source ~/.zshrc

# 프로젝트 폴더에서 Node 버전 지정 (파일로 저장됨)
mise use node@22
mise use python@3.12
💡 왜 brew install node 대신 mise를 써야 할까?
brew install node로 설치하면 버전이 하나로 고정된다. 여러 프로젝트를 다루거나 팀원과 같은 버전을 맞춰야 할 때 불편하다. mise를 쓰면 폴더에 들어가는 순간 자동으로 버전이 전환된다. 실리콘밸리 개발팀에선 거의 표준처럼 쓰인다.

1.3 터미널 환경 설정 (Zsh)

Mac의 기본 터미널 프로그램은 Zsh(제트쉬)다. 여기에 Oh My Zsh를 설치하면 자동완성, 문법 강조, 유용한 단축키 등 다양한 기능을 쉽게 추가할 수 있다 — 마치 빈 노트앱에 플러그인을 설치하는 것처럼.

# Oh My Zsh 설치
sh -c "$(curl -fsSL https://raw.githubusercontent.com/ohmyzsh/ohmyzsh/master/tools/install.sh)"

~/.zshrc 파일에 단축 명령어(alias)를 추가한다 — 긴 명령어를 짧게 줄여 쓰는 나만의 별명이다:

# 자주 쓰는 단축 명령어
alias ll="eza -la --icons"       # 파일 목록 보기 (아이콘 포함)
alias cat="bat"               # 파일 내용 보기 (색상 있음)
alias grep="rg"              # 코드 검색 (빠른 버전)
alias ..="cd .."             # 상위 폴더로 이동
alias ...="cd ../.."         # 두 단계 위로 이동

# Git 단축 명령어
alias gs="git status"
alias ga="git add"
alias gc="git commit"
alias gp="git push"

fzf로 이전에 입력한 명령어를 빠르게 검색하는 기능 활성화:

# .zshrc에 추가 (fzf 0.48.0 이상 권장)
eval "$(fzf --zsh)"

추천 Oh My Zsh 플러그인: ~/.zshrc의 plugins 배열을 찾아 수정한다.

plugins=(
  git                      # git 단축 명령어 (gst, gco 등) - 기본 내장
  zsh-autosuggestions      # 이전 명령어 자동 제안 (→ 키로 수락)
  zsh-syntax-highlighting  # 올바른 명령어는 초록색, 잘못된 것은 빨간색으로 표시
  z                        # 자주 가는 폴더를 이름 일부만 입력해도 이동 - 기본 내장
)
# zsh-autosuggestions 설치 (Oh My Zsh에 기본 내장 안 됨)
git clone https://github.com/zsh-users/zsh-autosuggestions \
  ${ZSH_CUSTOM:-~/.oh-my-zsh/custom}/plugins/zsh-autosuggestions

# zsh-syntax-highlighting 설치
git clone https://github.com/zsh-users/zsh-syntax-highlighting \
  ${ZSH_CUSTOM:-~/.oh-my-zsh/custom}/plugins/zsh-syntax-highlighting

1.4 터미널 앱 선택

Mac 기본 터미널(Terminal.app)보다 훨씬 강력한 대안이 있다:

  • iTerm2 — 창 분할, 프로필 관리, 안정성 검증. 무료. brew install --cask iterm2
  • Warp — AI 명령어 제안, 블록 단위로 출력 관리, 현대적 UI. brew install --cask warp
💡 팁
brew install --cask 앱이름 형태로 GUI 앱도 Homebrew로 설치할 수 있다. brew upgrade --cask로 설치한 앱들을 한 번에 업데이트하자. 앱스토어 없이 개발 도구를 통합 관리하는 것이 훨씬 편하다.

VS Code Mac 최적화

핵심 설정 · 필수 확장 · Mac 전용 단축키

VS Code는 세계에서 가장 많이 쓰이는 코드 편집기다. 기본 설정 그대로 써도 되지만, 몇 가지 설정만 바꿔도 개발 속도가 눈에 띄게 달라진다. 마치 사무실 의자 높이를 맞추는 것처럼 — 처음 한 번만 세팅하면 된다.

2.1 핵심 settings.json 설정

VS Code 안에서 ⌘⇧P를 누르고 "Open User Settings (JSON)"을 입력해 설정 파일을 열고, 아래 내용을 추가한다:

{
  "editor.fontSize": 14,
  "editor.fontFamily": "'JetBrains Mono', 'Fira Code', Menlo, monospace",
  "editor.fontLigatures": true,          // != 같은 기호를 이쁘게 합쳐서 표시
  "editor.tabSize": 2,
  "editor.formatOnSave": true,            // 저장할 때 자동으로 코드 정리
  "editor.minimap.enabled": false,         // 오른쪽 미니맵 끄기 (공간 확보)
  "editor.wordWrap": "on",
  "files.autoSave": "onFocusChange",      // 다른 창으로 이동하면 자동 저장
  "terminal.integrated.defaultProfile.osx": "zsh",
  "workbench.colorTheme": "One Dark Pro"
}

2.2 필수 확장 프로그램

⌘⇧X로 확장 마켓을 열거나, 터미널에서 아래 명령어로 한 번에 설치:

code --install-extension eamodio.gitlens
code --install-extension dbaeumer.vscode-eslint
code --install-extension esbenp.prettier-vscode
code --install-extension GitHub.copilot
code --install-extension usernamehw.errorlens
code --install-extension PKief.material-icon-theme
code --install-extension christian-kohler.path-intellisense
code --install-extension formulahendry.auto-rename-tag
확장하는 일
GitLens코드의 각 줄이 누가 언제 수정했는지 바로 보여줌
ESLint코드에서 실수나 규칙 위반을 실시간으로 표시해주는 검사기
Prettier저장할 때 코드를 자동으로 정해진 스타일에 맞게 정리
GitHub CopilotAI가 다음에 올 코드를 자동으로 제안해주는 도구
Error Lens에러 내용을 해당 코드 줄 바로 옆에 빨간 글씨로 표시
Material Icon Theme파일 종류에 맞는 아이콘으로 탐색기를 알아보기 쉽게
Path Intellisense파일 경로 입력할 때 자동완성으로 오탈자 방지
Auto Rename TagHTML 여는 태그를 바꾸면 닫는 태그도 자동으로 같이 변경
⚠️ Thunder Client 주의
Thunder Client(REST API 테스트 도구)는 2023년부터 팀 기능이 유료로 바뀌었다. 대신 Bruno(usebruno.bruno)를 추천한다. 파일 기반으로 동작해 Git으로 버전 관리가 되고, 팀 전체가 같은 API 요청 파일을 공유할 수 있다.

Claude Code 완전 정복

AI 에이전트 협업 · 슬래시 커맨드 · 실전 워크플로우

Claude Code(클로드 코드)는 터미널에서 AI에게 코딩 작업을 직접 맡길 수 있는 도구다. 단순히 "코드 써줘"를 넘어, 여러 AI 에이전트가 탐색·설계·구현을 역할을 나눠 협력하는 방식으로 복잡한 작업도 처리할 수 있다.

3.1 Claude Code란?

Claude Code는 코드베이스(프로젝트 전체 파일)를 읽고 이해한 상태에서 대화할 수 있다. 여러 전문 에이전트가 각자 역할을 맡아 협력하는 것이 핵심이다:

  • Explore 에이전트 — 코드베이스를 탐색하고 파일 구조를 파악하는 역할
  • Plan 에이전트 — 어떻게 구현할지 설계하고 작업 목록을 만드는 역할
  • Build 에이전트 — 실제 코드를 작성하고 파일을 수정하는 역할

3.2 핵심 슬래시 커맨드

이 프로젝트에서 사용하는 OpenSpec 연동 커맨드들 — /로 시작하는 명령어다:

커맨드하는 일
/opsx-explore탐색 모드 — 코드 수정 없이 분석·아이디어 탐구만 함
/opsx-propose변경 계획 생성 — 무엇을·어떻게·어떤 순서로 할지 문서 3개 자동 생성
/opsx-apply실제 구현 — 앞서 만든 계획에 따라 코드 작성
/opsx-archive완료 처리 — 구현 완료된 변경사항을 아카이브로 이동

3.3 실전 워크플로우

# 1단계: 탐색 — 문제 파악 (코드 수정 없음)
/opsx-explore 로그인 기능에 어떤 보안 문제가 있을 수 있을까?

# 2단계: 계획 — 어떻게 고칠지 설계
/opsx-propose improve-login-security

# 3단계: 구현 — 실제 코드 작성
/opsx-apply improve-login-security

# 4단계: 마무리 — 완료 처리
/opsx-archive improve-login-security

AI 에이전트에게 일을 맡길 때는 구체적인 배경 설명이 결과를 좌우한다:

# 좋은 예시: 어디서, 무슨 에러가, 왜 생기는지 설명
"src/auth/login.ts의 validateToken 함수에서
만료된 로그인 토큰을 처리할 때 에러가 catch되지 않아
서버가 500 에러를 반환하고 있어. 만료된 토큰이면
401(인증 실패) 응답을 보내도록 수정해줘."

# 나쁜 예시: 정보가 너무 적음
"로그인 버그 고쳐줘"
💡 팁
Claude Code는 여러 에이전트를 동시에 실행할 수 있다. 서로 연관이 없는 작업들 — 예를 들어 "파일 A 분석"과 "파일 B 분석" — 은 동시에 시킬 수 있어 시간을 절약한다. 또한 작업 도중 에이전트가 멈추면 같은 세션을 이어받아 앞서 파악한 내용을 유지하면서 계속 진행할 수 있다.
4

Open Spec — 설계서 기반 개발

먼저 설계하고, 그 다음 코딩하는 4단계 워크플로우

Open Spec은 "코드를 바로 짜지 말고, 먼저 무엇을 왜 어떻게 만들지 글로 정리한 다음 구현하자"는 개발 방식이다. 건물 짓기 전에 설계도를 그리는 것과 같다 — AI가 있으면 이 설계서 작성도 자동화된다.

4.1 왜 설계서 기반 개발인가

  • 의도가 명확해진다 — 코드를 보지 않아도 "왜 이렇게 만들었는지" 알 수 있다
  • AI 결과물이 좋아진다 — 설계서가 있으면 AI가 맥락을 파악하고 더 정확하게 구현한다
  • 리뷰가 쉬워진다 — 코드 리뷰 전에 설계 방향부터 검토할 수 있다
  • 히스토리가 남는다 — 6개월 뒤에 "이걸 왜 이렇게 만들었지?"라는 질문에 답이 된다

4.2 4단계 워크플로우

  1. 탐색 (Explore) — /opsx-explore [주제]
    코드를 건드리지 않고 현재 상태를 파악하는 단계. "이 기능 어떻게 돼있어?"
  2. 계획 (Propose) — /opsx-propose [변경이름]
    무엇을 만들지(proposal.md), 어떻게 만들지(design.md), 어떤 순서로(tasks.md) 세 문서가 자동 생성된다.
  3. 구현 (Apply) — /opsx-apply [변경이름]
    tasks.md의 체크박스를 하나씩 처리하며 실제 코드를 작성한다. 중단해도 어디까지 했는지 명확하다.
  4. 아카이브 (Archive) — /opsx-archive [변경이름]
    완료된 변경사항을 기록 보관소(openspec/specs/)로 옮겨 히스토리로 남긴다.

4.3 언제 쓰면 좋을까

✅ 새로운 기능 추가 — 범위와 목적을 미리 명확히 해야 할 때
✅ 복잡한 코드 정리 — 영향을 받는 부분을 먼저 파악해야 할 때
✅ 팀 협업 — 내가 뭘 만들려는지 다른 사람도 이해해야 할 때
✅ AI와 협업 — 설계서가 있으면 AI 결과물 품질이 크게 향상

❌ 간단한 버그 수정 — 굳이 설계서까지는 필요 없음
❌ 아직 방향이 안 잡힌 실험 — 너무 자주 방향이 바뀌면 설계서가 금방 낡아짐
💡 팁
openspec/config.yaml에 프로젝트 기술 스택과 코딩 규칙을 미리 써두면, AI가 이 프로젝트에 맞는 방식으로 설계서와 코드를 만들어준다. "TypeScript, React, 함수형 컴포넌트만 사용"처럼 짧게라도 써두는 것이 큰 차이를 만든다.

AI 프롬프트 작성법

5가지 패턴 · Before/After 비교 · 즉시 쓸 수 있는 템플릿

AI에게 일을 시킬 때 어떻게 말하느냐가 결과를 결정한다. 막연하게 물으면 막연한 답이 오고, 구체적으로 물으면 바로 쓸 수 있는 코드가 온다. 좋은 지시는 좋은 요리 레시피처럼 — 재료, 목표, 제약이 명확해야 한다.

6.1 상황별 프롬프트 패턴

패턴 1 버그 수정 — 어디서, 무슨 에러가, 왜 생기는지 알려주기

[파일 위치]: src/auth/login.ts
[함수 이름]: validateToken()
[에러 메시지]: TokenExpiredError: jwt expired
[예상 동작]: 만료된 토큰이면 401(인증 실패) 응답 반환
[실제 동작]: 500(서버 에러) 발생
수정해줘.

패턴 2 코드 정리 (리팩토링) — 목표와 지켜야 할 조건 명시

아래 코드를 읽기 쉽게 정리해줘.
[목표]: 중복 제거, 가독성 향상
[조건]: 외부에서 사용하는 API 형태는 바꾸지 않기, TypeScript strict 모드 유지
[코드]: (코드 붙여넣기)

패턴 3 테스트 코드 생성

아래 함수의 테스트 코드를 작성해줘.
[테스트 도구]: Vitest
[포함할 케이스]: 정상 동작, 엣지 케이스(극단적인 입력), 에러 상황
[함수]: (코드 붙여넣기)

패턴 4 새 기능 구현

[만들 기능]: 사용자 프로필 이미지 업로드

[요구사항]:

[참고할 기존 코드]: src/routes/user.ts (비슷한 패턴 사용)
구현해줘.

6.2 Before/After 비교

# ❌ 이렇게 물으면 AI도 막막함
"로그인이 안 돼요"

# ✅ 이렇게 물어야 바로 쓸 수 있는 답이 옴

6.3 프롬프트 품질을 높이는 4가지 원칙

  • 역할 부여 — "시니어 TypeScript 개발자로서..." — AI가 더 전문적인 관점으로 답변한다
  • 출력 형식 지정 — "코드 블록으로만 답해줘", "표로 비교해줘"
  • 제약 조건 명시 — "외부 라이브러리 추가 없이", "기존 API 형태 유지"
  • 예시 제공 — "이 파일과 같은 스타일로 작성해줘: (파일 경로)"
💡 팁
프롬프트 끝에 "단계별로 생각하면서 답해줘"를 붙이면 복잡한 문제에서 AI의 답변 품질이 크게 향상된다. 또 Raycast 스니펫(단축 입력 기능)에 자주 쓰는 프롬프트 템플릿을 등록해두면 반복 입력을 줄일 수 있다 — `:bugfix`를 입력하면 버그 수정 템플릿 전체가 펼쳐지는 식이다.
7

키보드 & 생산성 해킹

Rectangle · Raycast · Karabiner · 시스템 튜닝

Mac에서 진짜 생산성의 차이는 마우스를 얼마나 안 쓰느냐에서 나온다. 손이 키보드를 떠나지 않아도 되는 환경을 만들면, 흐름이 끊기지 않고 생각의 속도로 코딩할 수 있다.

7.1 창 관리 — Rectangle

Rectangle은 단축키로 창을 화면 절반, 전체, 1/4 등 원하는 위치에 딱 맞춰주는 무료 도구다. 마우스로 창 크기를 드래그하는 시간을 없애준다.

brew install --cask rectangle
단축키결과
⌃⌥←화면 왼쪽 절반으로 이동
⌃⌥→화면 오른쪽 절반으로 이동
⌃⌥↩전체 화면 최대화
⌃⌥C화면 정중앙에 배치

7.2 앱 실행기 — Raycast

Raycast는 Mac의 Spotlight(⌘Space 검색 기능)를 완전히 대체하는 강력한 실행기다. 앱 실행, 파일 검색, 클립보드 히스토리, 단축 입력 등 다양한 기능을 키보드 하나로 처리할 수 있다.

brew install --cask raycast
  • 기본 단축키 ⌘Space로 실행 (Spotlight 단축키를 다른 것으로 바꿔야 함)
  • 클립보드 히스토리 ⌘⇧V — 이전에 복사한 내용들을 목록으로 보고 재사용
  • 스니펫 — :bugfix처럼 짧은 코드를 입력하면 긴 프롬프트 템플릿이 자동 완성
  • 윈도우 관리 — Raycast Pro를 쓰면 Rectangle 없이도 창 관리 가능
  • 빠른 링크 — "gh 검색어" 입력 시 GitHub 바로 검색, "mdn fetch" 입력 시 MDN 문서 바로 열기
  • 계산기·단위 변환 — Raycast 창에 바로 수식을 입력하면 즉시 계산
  • AI 기능 (Claude, ChatGPT 연동) 내장

7.3 키 재배치 — Karabiner-Elements

Karabiner-Elements는 Mac 키보드의 키 동작을 자유롭게 바꿀 수 있는 도구다. 가장 인기 있는 설정은 Caps Lock 키를 두 가지로 활용하는 것이다:

brew install --cask karabiner-elements
  • 짧게 누르면 Escape — Vim이나 터미널에서 매우 자주 쓰는 키를 손가락이 자연스러운 위치에
  • 길게 누르면 Hyper 키 (Ctrl+Option+Cmd+Shift 동시 입력) — 이 조합을 단독으로 쓰는 앱이 없으므로, Raycast나 다른 도구의 글로벌 단축키로 자유롭게 할당 가능
💡 Hyper 키가 왜 유용한가?
Ctrl+Option+Cmd+Shift를 동시에 사용하는 앱은 사실상 없다. 이 조합을 "Hyper 키"로 정의하면 충돌 걱정 없이 50개 이상의 나만의 전역 단축키를 만들 수 있다. Karabiner 설정 후 Raycast에서 이 조합을 원하는 동작에 연결하면 된다.

7.5 한영 전환 & Mission Control

기본 한영 전환 단축키(⌃Space)는 다른 앱의 단축키와 자주 충돌한다. Caps Lock으로 한영 전환하는 것이 훨씬 자연스럽다:

시스템 설정 → 키보드 → 입력 소스 → "Caps Lock 키로 ABC 전환" 활성화

단축키기능
⌃↑Mission Control — 열린 모든 창을 한눈에 보기
⌃↓현재 앱의 모든 창 보기
⌃→ / ⌃←가상 데스크탑(Spaces) 전환

Spaces(가상 데스크탑) 활용법 — 작업 종류별로 화면을 분리하면 집중력이 높아진다:

  • Space 1: 코딩 (VS Code, 터미널)
  • Space 2: 참고 자료 (브라우저, 문서)
  • Space 3: 소통 (Slack, 이메일)

7.6 macOS 시스템 튜닝 (defaults write)

터미널에서 아래 명령어를 실행하면 macOS의 숨겨진 설정을 바꿀 수 있다. 개발자들이 가장 많이 쓰는 튜닝 모음이다:

# Finder 제목 표시줄에 전체 경로 표시
defaults write com.apple.finder _FXShowPosixPathInTitle -bool true

# 키 반복 속도 최대화 (방향키, hjkl 이동이 훨씬 빨라짐)
defaults write NSGlobalDomain KeyRepeat -int 1
defaults write NSGlobalDomain InitialKeyRepeat -int 10

# 숨김 파일 표시 (앞에 . 붙은 파일들, 예: .zshrc, .gitconfig)
defaults write com.apple.finder AppleShowAllFiles YES

# 네트워크 드라이브에 .DS_Store 파일 생성 안 함
defaults write com.apple.desktopservices DSDontWriteNetworkStores -bool true

# 설정 적용을 위해 Finder 재시작
killall Finder
💡 팁
키 반복 속도 튜닝이 의외로 체감이 크다. 기본값은 방향키를 누르고 있어도 커서가 느리게 이동한다. KeyRepeat -int 1로 바꾸면 Vim 사용자나 방향키로 코드를 탐색하는 사람 모두 훨씬 빠르게 이동할 수 있다.
8

Git + AI 연동

커밋 메시지 자동화 · AI 코드 리뷰 · 충돌 해결

Git 작업에 AI를 연결하면 커밋 메시지 작성, 코드 리뷰 준비, 충돌 해결처럼 반복적으로 시간이 걸리는 작업을 크게 줄일 수 있다.

8.1 AI로 커밋 메시지 자동 작성

변경 내역(diff)을 AI에게 전달하면 형식에 맞는 커밋 메시지를 만들어준다. pbcopy는 터미널 출력을 클립보드로 복사하는 Mac 명령어다:

# 현재 저장(staged)된 변경사항을 클립보드에 복사
git diff --staged | pbcopy
# → 클립보드 내용을 AI에 붙여넣고 커밋 메시지 요청
아래 코드 변경사항을 보고 커밋 메시지를 작성해줘.
형식: <type>(<범위>): <한 줄 설명>
type 종류: feat(새 기능), fix(버그 수정), docs(문서), refactor(코드 정리), chore(기타)
[diff 붙여넣기]

8.2 PR 올리기 전 AI 코드 리뷰

팀원에게 PR 리뷰를 요청하기 전에 AI에게 먼저 검토받으면 기본적인 실수를 미리 잡을 수 있다:

# main 브랜치와의 전체 변경사항을 클립보드에 복사
git diff main...feature/my-feature | pbcopy
아래 코드 변경사항을 리뷰해줘.
확인 항목:
1. 논리 오류나 버그
2. 보안 취약점 (SQL 인젝션, XSS 등)
3. 성능 문제
4. 코드 스타일 일관성
5. 테스트가 없는 중요한 로직
[diff 붙여넣기]

8.3 머지 충돌 해결 & Git Alias

AI로 머지 충돌 해결:

# 충돌이 난 파일을 클립보드에 복사
cat conflicted-file.ts | pbcopy
아래 파일에 머지 충돌이 있어.
<<<<<<< HEAD 부분은 현재 내 코드,
>>>>>>> feature 부분은 합치려는 코드야.
두 변경사항을 올바르게 합쳐줘.
[파일 내용 붙여넣기]

자주 쓰는 Git 단축 명령어 모음:

# ~/.gitconfig에 추가
[alias]
  lg     = log --oneline --graph --decorate --all   # 브랜치 히스토리 시각화
  st     = status -sb                               # 변경사항 요약 보기
  undo   = reset HEAD~1 --mixed                     # 마지막 커밋 취소 (파일은 유지)
  save   = stash push -m                            # 작업 임시 저장
  recent = branch --sort=-committerdate             # 최근 브랜치 목록
💡 팁
git log --oneline -20 | pbcopy로 최근 커밋 목록을 복사한 뒤 AI에게 "이 프로젝트의 커밋 메시지 스타일을 분석해줘"라고 요청하면, 기존 팀 컨벤션에 맞는 커밋 메시지를 생성해준다. 새 팀에 합류했을 때 특히 유용하다.
9

AI로 버그 잡기

에러 분석 · AI 고무 오리 디버깅 · VS Code 디버거

버그 하나에 혼자 몇 시간씩 씨름하는 것보다 AI에게 먼저 물어보는 게 훨씬 효율적이다. 에러 메시지와 관련 코드를 그대로 붙여넣으면 대부분 빠르게 원인을 찾을 수 있다.

9.1 에러 발생 시 기본 템플릿

[에러 메시지]:
TypeError: Cannot read properties of undefined (reading 'map')

[에러 위치 (스택 트레이스)]:
at UserList (src/components/UserList.tsx:15:23)

[관련 코드]:
const UserList = ({ users }) => {
  return users.map(user => <div>{user.name}</div>)
}

[발생 상황]: 페이지 처음 로드할 때만 발생
[이미 시도한 것]: users가 undefined인지 확인했지만 원인 불명

원인과 해결 방법을 알려줘.
# 빌드 에러를 바로 클립보드로 복사하는 방법
npm run build 2>&1 | pbcopy

9.2 AI 고무 오리 디버깅

"고무 오리 디버깅"은 막혔을 때 문제를 누군가에게 설명하다 보면 스스로 답을 찾는 현상을 말한다. AI에게 설명하는 것도 똑같이 효과적이다:

지금 디버깅하는 문제를 설명할게.
내가 이해하는 것과 이해 못 하는 것을 정리해줘.

[문제 상황]: 사용자가 로그아웃 후에도 일부 API 요청이 성공함
[시도한 것]: 토큰 삭제 코드 확인 — 정상적으로 삭제됨
[현재 추측]: 토큰이 서버 캐시에 남아있는 게 아닐까?

내 추측이 맞을까? 놓친 게 있을까?

9.3 VS Code 디버거 설정

터미널 에러만으로 부족할 때는 VS Code 디버거를 쓴다. 코드 특정 줄에 중단점을 설정하면 그 시점의 변수 값을 직접 확인할 수 있다:

// .vscode/launch.json — Node.js TypeScript 프로젝트 기준
{
  "version": "0.2.0",
  "configurations": [{
    "type": "node",
    "request": "launch",
    "name": "디버그 실행",
    "program": "${workspaceFolder}/src/index.ts",
    "runtimeArgs": ["-r", "ts-node/register"],
    "sourceMaps": true,
    "envFile": "${workspaceFolder}/.env"
  }]
}

AI 디버깅 반복 패턴:

  1. 에러 발생 → 에러 메시지 + 에러 위치 로그 복사
  2. AI에게 "원인이 뭘까?" 질문
  3. AI 제안대로 코드 수정 후 테스트
  4. 여전히 안 되면 → "이 방법은 안 됐어. 다른 접근은?" 계속 대화
  5. 해결 후 → "이 버그의 근본 원인이 뭔지, 앞으로 어떻게 예방하면 되는지 설명해줘"
💡 팁
AI에게 "이 코드에서 버그 찾아줘"보다 "이 코드가 X 상황에서 Y 결과를 내야 하는데 Z가 나와. 왜 그럴까?"처럼 기대하는 결과와 실제 결과를 함께 알려주면 훨씬 정확한 진단을 받을 수 있다.
10

개발 환경 유지보수

Homebrew 업데이트 루틴 · dotfiles 관리 · 디스크 정리

개발 환경도 주기적으로 관리해야 한다. 방치하면 패키지가 낡아지고, 디스크가 꽉 차고, macOS 업데이트 후 환경이 깨지는 상황이 생긴다. 작은 루틴 하나가 큰 문제를 예방한다.

10.1 Homebrew 업데이트 루틴

주 1회 정도 실행을 권장한다. 단축 명령어(alias)로 등록해두면 편하다:

# 한 번에 업데이트 + 업그레이드 + 불필요한 파일 정리
brew update && brew upgrade && brew cleanup

# 문제 진단 — 뭔가 이상하면 실행
brew doctor

# .zshrc에 단축 명령어로 등록
alias brewup="brew update && brew upgrade && brew cleanup && brew doctor"

10.2 dotfiles — 설정 파일 백업 및 동기화

dotfiles는 .zshrc, .gitconfig 같은 설정 파일들을 말한다 (파일명이 .으로 시작해서 dotfiles). 이것들을 Git으로 관리하면 새 Mac을 받았을 때 10분 안에 똑같은 환경을 복원할 수 있다.

chezmoi로 dotfiles 관리하기 — 여러 Mac에서 같은 설정을 쓰거나, 회사 Mac과 개인 Mac 설정을 조금씩 다르게 유지하고 싶다면 chezmoi(쉐모아)를 쓴다:

# chezmoi 설치
brew install chezmoi

# 초기화 (GitHub에 dotfiles 저장소가 있다면)
chezmoi init --apply https://github.com/내아이디/dotfiles

# 설정 파일 등록
chezmoi add ~/.zshrc
chezmoi add ~/.gitconfig

# 변경사항 GitHub에 업로드
chezmoi cd
git add . && git commit -m "update zshrc" && git push
# .gitconfig 기본 설정 예시
[user]
  name  = 이름
  email = 이메일@example.com
[core]
  editor = code --wait   # Git 에디터로 VS Code 사용
[alias]
  lg = log --oneline --graph --decorate

10.3 디스크 정리 & macOS 업데이트 대응

# 프로젝트 폴더의 모든 node_modules 삭제 (재설치 필요하지만 수GB 확보)
find ~/projects -name "node_modules" -type d -prune -exec rm -rf '{}' +

# 패키지 캐시 정리
bun pm cache rm
brew cleanup --prune=all

macOS 업데이트 후 체크리스트:

brew doctor              # Homebrew 정상 동작 확인
node --version           # Node.js 버전 확인
git --version            # Git 정상 동작 확인
xcode-select --install   # macOS 업데이트 후 개발 도구 재설치 (필요시)
💡 팁
brew bundle dump를 실행하면 현재 설치된 모든 Homebrew 도구 목록이 Brewfile로 저장된다. 이 파일을 dotfiles 저장소에 함께 관리하면, 새 Mac에서 brew bundle install 한 줄로 모든 도구를 한 번에 복원할 수 있다. 환경 복원이 10분으로 줄어든다.

마치며

10가지를 모두 한 번에 적용하려 하지 말자. 지금 당장 가장 불편한 것 하나부터 시작하면 된다. 처음 Mac을 세팅한다면 챕터 1부터, AI 도구를 더 잘 활용하고 싶다면 챕터 3과 챕터 4부터, 오늘 당장 키보드 단축키 몇 개라도 외우고 싶다면 챕터 7부터 시작하자.

특히 Claude Code와 Open Spec의 조합은 AI 코딩 워크플로우의 핵심이다. 설계서를 먼저 쓰고, AI가 그 설계서를 따라 구현하는 이 방식은 — 단순히 "AI가 코드를 써준다"를 넘어, AI와 협업하는 구조를 만드는 것이다.

AI 도구는 빠르게 발전하고 있다. 중요한 건 특정 도구에 집착하지 않고 항상 더 효율적인 방법을 찾는 자세다. 이 가이드가 그 탐색의 좋은 출발점이 되길 바란다.
